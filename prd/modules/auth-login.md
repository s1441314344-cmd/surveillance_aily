# 登录页

## 1. 需求概要

### 模块定位
登录页是智能巡检系统 V2 的统一入口，负责建立用户会话，并把用户导向有权限访问的工作台。

### 目标
- 提供统一登录入口
- 建立可刷新的本地会话
- 在权限不足时给出明确的拒绝访问反馈

### 页面范围
- 用户名密码登录
- 登录成功后的跳转
- 登录失败反馈

### 模块依赖
- **全局规则**：`docs/PRD/需求总览.md` §3 角色与权限、§4.4 事实源优先级
- **关联模块**：`docs/PRD/modules/users-and-permissions.md` §5 页面数据模型

### Demo 实现状态

| 功能点 | Demo 状态 | 数据来源 | 说明 |
|--------|-----------|----------|------|
| 用户名密码登录 | ✅ 已实现 | `frontend-v2` + `backend-v2` | 默认开发账号可直接登录 |
| 登录成功跳转 | ✅ 已实现 | 前端路由 | 成功后进入 `/dashboard` |
| 拒绝访问提示 | ✅ 已实现 | 前端路由守卫 | 由 `AccessDeniedPage` 承接 |

### 一期不做
- 当前不做多因子认证
- 当前不做企业 SSO 正式接入

## 2. 页面功能总览
- 登录表单：采集用户名和密码并发起会话建立请求
- 会话建立：保存 access token、refresh token 和当前用户信息
- 访问引导：成功后跳转到总览页，失败时给出明确错误反馈

## 3. 功能模块

### 3.1 登录提交
#### 3.1.1 功能目的
让有权限的用户进入系统主工作台。

#### 3.1.2 触发条件
用户在登录页填写用户名和密码并点击“登录系统”。

#### 3.1.3 前置条件
本功能无独立前置条件。

#### 3.1.4 主流程
1. 用户输入用户名和密码。
2. 系统校验必填项是否完整。
3. 表单提交后进入加载中状态，并向认证服务发起登录请求。
4. 登录成功时，系统保存当前会话信息并写入本地会话状态。
5. 页面跳转到总览看板。

#### 3.1.5 流程图
本功能逻辑线性、无复杂分支，省略流程图。

#### 3.1.6 状态与反馈
- 默认态：展示默认账号提示和空表单
- 加载中：登录按钮进入不可重复提交状态
- 成功：提示“登录成功”并跳转
- 失败：提示登录失败原因

#### 3.1.7 交互与展示
- 登录表单包含用户名、密码两个字段
- 页面展示默认开发账号说明，便于联调
- 提交按钮全宽展示，强调这是统一入口页

#### 3.1.8 异常与边界
- 必填项为空时禁止提交
- 登录请求进行中时禁止重复点击
- 认证失败时保留当前输入，不自动清空

### 3.2 会话保持与刷新
#### 3.2.1 功能目的
让已登录用户在 token 过期后尽量无感续期。

#### 3.2.2 触发条件
已登录用户访问受保护页面，且当前 access token 过期。

#### 3.2.3 前置条件
本地存在可用 refresh token。

#### 3.2.4 主流程
1. 受保护请求返回未认证结果。
2. 系统尝试使用 refresh token 续期。
3. 续期成功时更新本地会话并重放原请求。
4. 续期失败时清空本地会话并回到登录页。

#### 3.2.5 流程图
本功能逻辑线性、无复杂分支，省略流程图。

#### 3.2.6 状态与反馈
- 成功：用户无感继续访问
- 失败：退出登录并重新登录

#### 3.2.7 交互与展示
- 会话刷新过程不额外展示独立弹窗
- 登录过期后的跳转应保持可理解的路径

#### 3.2.8 异常与边界
- refresh token 缺失时直接退出登录
- 同时触发多个刷新请求时只允许一个真实刷新流程

## 4. 页面级共享规则
- 登录成功默认进入 `/dashboard`
- 拒绝访问不回到登录页，而是通过“无权限访问”页处理

## 5. 数据模型

### 5.1 依赖共享实体
- `User / Role`：引用 `docs/PRD/需求总览.md` §3

### 5.2 页面私有数据
| 字段 | 含义 |
|------|------|
| `username` | 登录用户名 |
| `password` | 登录密码 |
| `token` | 当前 access token |
| `refresh_token` | 当前 refresh token |
| `display_name` | 当前用户显示名 |
| `roles` | 当前用户角色集合 |
