# 智能巡检系统 V2 需求总览

## 1. 文档定位

### 1.1 文档目的
本文件是智能巡检系统 V2 的总览 PRD，用来统一定义产品范围、角色权限、共享实体、状态口径和页面地图，作为各页面模块 PRD 的上游口径。

### 1.2 当前使用边界
- 本文件只定义业务语义、范围边界、共享实体、状态、指标口径和路由映射。
- 本文件不定义 HTTP 端点、请求体、数据库表结构和技术实现细节。
- 历史 `docs/product`、`docs/plan`、`docs/architecture` 仍保留参考价值，但当前 PRD 口径以本文件和 `docs/PRD/modules/*.md` 为准。

## 2. 产品定位与范围

### 2.1 产品定位
智能巡检系统 V2 是面向图片上传与摄像头巡检场景的独立版智能巡检平台，主交付面是 Web 工作台、独立 API 服务和本地检测侧车服务。

### 2.2 核心目标
- 统一管理巡检策略、模型配置、摄像头设备和异步任务执行。
- 把图片/摄像头输入转成可追溯的任务记录和结构化结果。
- 建立人工复核、统计看板和治理能力，避免结果只停留在单次检测页面。
- 在不破坏现有 V2 主链的前提下，持续收敛文档、代码和测试口径。

### 2.3 当前范围内
- 登录与角色权限
- 总览看板与看板配置
- 策略中心
- 摄像头中心索引入口与四个子页面
- 告警中心
- 任务中心
- 任务记录
- 人工复核
- 模型与系统设置
- 用户与权限
- 操作审计
- 本地检测调试

### 2.4 当前范围外
- `mes-workflow-designer` 及其相关原型
- 旧飞书 / Aily 主业务流
- 大规模数据库重构或服务拆分
- 把 ONVIF 作为当前主执行通道
- 把拖拽式看板编排升级为本轮必须落地能力

### 3.1 权限规则
- 导航和路由权限以当前代码中的单一 route registry 为事实基础。
- `任务记录` 已按受控页面处理，允许角色固定为 `system_admin`、`task_operator`、`manual_reviewer`、`analysis_viewer`。
- 模型密钥、敏感配置、用户管理、摄像头治理和本地检测只允许系统管理员访问，策略配置员只可进入策略中心和模型与系统设置。
- 页面级权限判定由模块 PRD 具体声明，但不得违背本节全局原则。

## 4. 全局规则与共享状态

### 4.1 主业务闭环
主业务闭环固定为：

`策略配置 -> 图片上传 / 摄像头输入 -> 异步任务 -> 任务记录 -> 人工复核 -> 总览统计`

### 4.2 主线状态枚举
| 对象 | 状态 |
|------|------|
| 策略 | `active` / `inactive` |
| 模型提供方 | `active` / `inactive` |
| 任务 | `queued` / `running` / `completed` / `failed` / `cancelled` |
| 定时计划 | `active` / `paused` |
| 任务记录 | `completed` / `failed` / `schema_invalid` |
| 反馈状态 | `unreviewed` / `correct` / `incorrect` |
| 摄像头连接状态 | `unknown` / `online` / `offline` / `warning` |
| 摄像头告警状态 | `normal` / `warning` / `error` |
| 告警事件 | `open` / `acked` / `resolved` |

### 4.3 扩展子域定级
- 告警中心：已存在的治理 / 运营子域，不再视为未来规划。
- 训练回流：已存在的扩展子域，挂在“模型与系统设置”页面，不纳入本轮主闭环核心链路。
- 本地检测：治理 / 调试子域，不直接承担正式归档职责。
- 版本识别：独立扩展流程，当前不纳入页面 PRD 主列表。

### 4.4 事实源优先级
- 当前代码事实优先于历史规划文本。
- 若代码与旧文档冲突：
  - 安全、权限、状态、范围以收紧口径为准。
  - 已实现的页面和对象要在 PRD 中被正名。
  - 未真正落地的能力降级为“未纳入当前主链”。

## 5. 共享实体与统计口径

### 5.1 共享实体
- `User / Role`
- `AnalysisStrategy / StrategyVersion`
- `ModelProvider`
- `Camera / CameraStatusLog`
- `Job / JobSchedule`
- `TaskRecord / PredictionFeedback`
- `DashboardDefinition`
- 支撑实体：`FileAsset`、`CameraMedia`、`ModelCallLog`、`OperationAuditLog`
- 告警扩展实体：`AlertEvent`、`AlertWebhookEndpoint`、`AlertNotificationRoute`
- 训练回流扩展实体：`FeedbackTrainingDataset`、`FeedbackTrainingRun`、`FeedbackReleaseRequest`
- 本地检测扩展实体：本地检测配置、本地检测结果

### 5.2 指标口径
- 总任务数：统计筛选范围内的 `Job`
- 总记录数：统计筛选范围内的 `TaskRecord`
- 结构化成功率：`normalized_json` 成功落库且结果状态不为 `schema_invalid` 的记录占比
- 异常率：命中异常类型或失败状态的记录占比
- 待复核数：`feedback_status = unreviewed` 的记录数
- 复核准确率：在已复核记录中，`judgement = correct` 的占比

### 5.3 非功能约束
- 系统是 API + worker + scheduler + PostgreSQL + Redis + `local-detector` 的异步链路，不按单进程 CRUD 产品定义。
- 当前发布与回归仍受旧数据回灌和脚本门禁约束，不能在文档里假设这是一个无历史负担的新系统。
- 主链优化不得回退现有 `task_dispatcher`、`job_service / scheduler_service facade` 和 `/api/*` 主契约边界。

## 7. 当前版本边界与阶段建议

### 7.1 当前阶段重点
- 主链先收敛“策略 -> 摄像头 / 任务 -> 记录 -> 复核 -> 看板”的正式口径。
- 告警中心、训练回流、本地检测虽然属于扩展 / 治理子域，但已经进入正式页面清单，不再按“未来规划”处理。
- 代码优化优先围绕权限、路由、共享状态、领域 API 边界和页面结构收敛，不扩新功能。

### 7.2 后续再做
- 拖拽式看板编排
- 复杂告警编排与跨团队通知模板中心
- 复杂训练编排设计器
- 大规模任务编排和跨策略批量助手

## 8. 开放问题
- 版本识别是否需要正式进入主导航，目前保持为扩展流程。
