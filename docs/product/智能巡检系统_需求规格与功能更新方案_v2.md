## 1. 文档目的

### 1.1 基线治理信息

- **基线生效**：自 2026-04-04 起，V2 需求、Backlog、验收、自动化证据口径以本文档与追踪矩阵为唯一基准。
- **冻结条件**：出现以下任一条件时，需求基线冻结，新增需求仅可进入后续版本：
  - P0 链路（策略 -> 任务 -> 记录 -> 反馈 -> 看板）验收条目全部通过。
  - 对应 P0 Backlog 项关闭且证据归档路径完整。
  - 追踪矩阵中 FR 到验收与自动化映射无空洞。
- **变更门禁**：冻结后修改 FR 或接口命名，需由 PM 与 ARCH 联合审批，并同步更新追踪矩阵与验收清单。

本文档用于重建智能巡检系统的 V2 需求基线，统一当前仓库中存在的以下偏差：

- 旧版文档仍以飞书/Aily 与 YOLO 本地识别为中心
- 现有独立版 Web 工作台已演进为任务驱动、多模型分析、记录沉淀的系统形态
- 规则、提示词模板、任务、记录、工单、飞书端能力之间存在边界混用
- 模型选型、JSON 输出要求、摄像头能力与任务状态缺少正式定义

V2 文档采用“任务驱动”产品模型，固定主闭环为：

`图片获取 -> 分析策略 -> 大模型分析 -> 任务队列 -> 任务记录 -> 人工反馈 -> 报告分析`

## 2. 产品定义

### 2.1 产品名称

统一名称为：**智能巡检系统 V2**

### 2.2 产品目标

面向图像上传与摄像头巡检场景，提供以下核心能力：

- 多源图片输入与采集
- 面向场景的分析策略管理
- 多模型视觉分析与标准化 JSON 输出
- 异步任务化执行与状态跟踪
- 结果留痕、人工反馈、报表分析
- 可持续演进的模型评估与迁移能力

### 2.3 产品定位

本产品是一个**独立版智能巡检平台**，以本地 Web 管理界面和独立 API 服务为主，不再以 Feishu Base、Aily 技能链路或飞书小程序为默认主流程。

### 2.4 范围界定

**In Scope**

- 独立 Web 平台
- 图片上传与摄像头抽帧
- 分析策略管理
- 大模型视觉分析
- 结构化 JSON 输出
- 任务队列与任务记录
- 人工反馈标注
- 基础统计与报表分析

**Out of Scope**

- `mes-workflow-designer` 及其原型
- 旧版 MES 工作流相关需求
- 将飞书/Aily 作为主业务依赖的方案

**Legacy Scope**

- Feishu/Aily 接入链路
- 飞书网页应用
- 飞书小程序
- 旧版定时调度和关键帧上报逻辑

上述遗留能力仅保留兼容说明，不纳入 V2 主需求中心。

### 2.5 版本策略

**V1**

- 单张/批量图片上传
- RTSP 摄像头抽帧
- 分析策略管理
- 模型提供方配置
- JSON 标准输出
- 任务队列与任务记录
- 基础统计看板
- 人工反馈标记

**V2**

- ONVIF 扩展
- 拖拽式看板
- 自定义图表组合
- 异常案例复核工作台
- 识别准确率趋势
- 遗留兼容能力治理

## 3. 用户角色与权限

V2 仅定义 RBAC 需求，不绑定具体认证厂商。实现上默认预留本地登录或企业 SSO 扩展点。

| 角色 | 核心职责 | 关键权限 |
|---|---|---|
| 系统管理员 | 管理模型、全局设置、摄像头、策略、用户权限 | 全量读写、敏感配置管理 |
| 策略配置员 | 维护分析策略、JSON Schema、场景模板 | 策略与模板读写 |
| 任务操作员 | 提交任务、查看任务状态、查看结果 | 任务创建、记录只读 |
| 人工复核员 | 复核模型结论、标记正确性、处理异常案例 | 反馈写入、记录详情读 |
| 分析查看者 | 查看统计与趋势报表 | 报表只读、记录汇总只读 |

### 3.1 权限原则

- 模型密钥、认证信息、用户权限仅系统管理员可见
- 策略配置员不可直接查看未脱敏的模型密钥
- 任务操作员不可修改系统级模型配置
- 人工复核员可标记反馈，但不可修改策略定义
- 报表查看者只能访问聚合数据和授权范围内的任务记录

| 场景编号 | 场景名称 | 说明 |
|---|---|---|
| US-01 | 手动上传分析 | 用户上传一张或多张图片，选择策略并提交模型分析 |
| US-02 | 摄像头单次抽帧 | 用户选择摄像头与策略，立即抽帧并生成一次任务 |
| US-03 | 摄像头定时巡检 | 用户创建定时任务，系统周期性抽帧、分析并记录结果 |

| US-07 | 模型选型评估 | 研发或管理员对比 OpenAI 与智谱模型在真实样本集上的表现 |

## 5. 业务流程

### 5.1 主流程

1. 管理员配置模型提供方与全局参数
2. 策略配置员创建或维护分析策略
3. 管理员配置摄像头及抽帧参数
4. 任务操作员通过上传图片或选择摄像头发起任务
5. 系统将任务写入任务队列并进入状态流转
6. 模型执行分析并返回标准化 JSON
7. 系统保存任务记录、原始响应和结构化结果
8. 人工复核员对结果进行正确性标记
9. 报表模块基于任务记录与反馈数据生成统计结果

### 5.2 任务状态流转

任务状态统一定义为：

- `queued`：已提交，等待执行
- `running`：执行中
- `completed`：执行完成且成功产出标准结果
- `failed`：执行失败
- `cancelled`：被用户或系统取消

### 5.3 关键约束

- 所有正式任务都必须先写入任务表，再进入执行
- 同步检测接口仅保留为快速预览能力，不作为正式主流程
- 正式任务的核心归档对象是 `TaskRecord`，而不是单次临时检测结果

## 8. 功能需求

### A. 图片获取模块

#### FR-A01 文件上传

系统应支持通过文件上传导入图片，支持单张上传与批量上传。

#### FR-A02 支持格式

系统应支持以下常见图片格式：

- JPG
- JPEG
- PNG
- BMP
- WEBP

#### FR-A03 文件校验

系统应在任务创建前校验图片格式、大小、可读性与损坏情况，并给出明确错误提示。

#### FR-A04 摄像头单次抽帧

系统应支持用户选择摄像头后执行一次即时抽帧，并将抽帧结果作为任务输入。

#### FR-A05 摄像头定时抽帧

系统应支持按频率定时抽帧，频率需可配置。

#### FR-A06 抽帧参数

系统应支持配置以下抽帧参数：

- 抽帧频率
- 分辨率
- JPEG 质量
- 存储路径

#### FR-A07 协议支持

V1 正式支持 `RTSP`；`ONVIF` 作为 V2 扩展能力。

#### FR-A08 采集稳定性

系统应支持抓帧失败重试，并记录最近失败原因和失败时间。

### B. 大模型集成与策略配置模块

#### FR-B01 策略管理

系统应提供分析策略管理界面，支持创建、编辑、删除、启用、禁用。

#### FR-B02 策略字段

每个策略必须包含：

- 策略名称
- 应用场景描述
- 提示词模板
- 模型提供方
- 模型名称
- JSON 返回格式定义
- 启用状态
- 版本号

#### FR-B03 预设场景

系统应预置至少以下典型场景策略：

- 安全帽识别
- 火情识别
- 瓶盖日期识别

#### FR-B04 模型提供方配置

系统应支持管理多个模型提供方，至少覆盖：

- 智谱
- OpenAI

#### FR-B05 JSON 标准输出

模型正式返回结果必须采用 JSON 格式，不允许仅依赖自然语言文本解析。

#### FR-B06 Schema 约束

每个策略应支持定义 JSON Schema，并用于响应校验。

#### FR-B07 结构化失败处理

当模型返回非 JSON 或不符合 Schema 时，系统应将任务标记为 `schema_invalid` 或 `failed`，并保留原始响应。

#### FR-B08 策略版本

任务执行时应保存策略快照，避免策略后续修改影响历史追溯。

#### FR-B09 模型评估能力

系统需求中必须保留模型评估章节，用于对比 OpenAI 与智谱在视觉识别场景中的表现。

### C. 摄像头配置模块

#### FR-C01 多摄像头管理

系统应支持添加、编辑、删除多个摄像头设备。

#### FR-C02 基础信息

摄像头需支持记录：

- 设备名称
- 位置信息
- 内网 IP
- 端口
- 协议
- 认证信息

#### FR-C03 抽帧参数配置

摄像头配置需支持：

- 抽帧频率
- 图像分辨率
- 图像质量
- 存储路径

#### FR-C04 连接状态

系统应展示摄像头当前连接状态、最近检查时间和最近错误。

#### FR-C05 异常告警

当摄像头不可达或抓帧失败时，系统应显示异常状态与告警提示。

#### FR-C06 状态接口

系统应提供摄像头状态查询接口，用于前端轮询显示。

### D. 任务提报模块

#### FR-D01 手动上传任务

系统应支持用户通过上传单张或批量图片，选择分析策略后创建任务。

#### FR-D02 摄像头任务创建

系统应支持用户选择摄像头、分析策略、执行方式创建任务：

- 单次执行
- 定时执行

#### FR-D03 异步任务化

正式主流程应采用异步任务机制，任务创建成功后进入队列执行。

#### FR-D04 任务状态展示

系统应显示任务状态、模型名称、耗时、失败原因。

#### FR-D05 任务取消

系统应支持取消未完成任务。

#### FR-D06 批量任务

批量图片上传应被建模为一个任务，支持逐项统计已完成与失败数量。

#### FR-D07 快速预览

现有同步检测能力可保留为快速预览，但不作为正式可追踪任务的唯一方式。

#### FR-D08 人工反馈入口

任务结果详情应支持人工标记正确性。

### E. 任务记录模块

#### FR-E01 记录归档

系统应为每次正式任务保存完整记录。

#### FR-E02 记录字段

每条记录至少保存：

- 任务 ID
- 提交时间
- 使用策略
- 输入图片
- 模型原始响应
- 标准化 JSON
- 执行状态
- 执行来源

#### FR-E03 查询筛选

系统应支持按以下条件查询任务记录：

- 时间范围
- 策略类型
- 任务状态
- 摄像头
- 模型提供方

#### FR-E04 详情展示

系统应支持查看历史任务详情，包括：

- 原始图片
- 结果预览图
- 完整 JSON
- 原始模型返回
- 反馈标记结果

#### FR-E05 导出

V1 应支持导出 CSV；Excel 导出为 Should。

### F. 报告分析模块

#### FR-F01 基础看板

V1 应提供基础统计看板，至少展示：

- 正常/异常识别比例
- 各策略使用频率
- 任务成功率
- 模型结构化成功率

#### FR-F02 反馈驱动指标

识别准确率、复核趋势等指标必须基于任务记录与人工反馈，不得仅以模型返回结果推断。

#### FR-F03 异常案例列表

V1 应提供异常案例汇总视图，便于人工查看和复核。

#### FR-F04 拖拽式看板

V2 应支持拖拽式报表编排和多图表组合。

#### FR-F05 趋势分析

V2 应支持准确率趋势、策略趋势、模型效果趋势展示。

### G. 非功能需求

#### FR-G01 性能

- 图片上传校验响应时间目标 `< 1s`
- RTSP 建连目标 `< 5s`
- 任务创建响应目标 `< 500ms`
- 模型分析 SLA 分为“提交成功”和“任务完成”两层

#### FR-G02 可靠性

- 摄像头抓帧失败可重试
- 任务失败不导致服务崩溃
- 定时任务需记录最近错误与下次执行时间

#### FR-G03 安全

- 需要鉴权和 RBAC
- 模型密钥必须脱敏显示
- 文件访问接口需限制路径范围与访问权限
- 关键操作应保留审计信息

#### FR-G04 兼容性

- V1 浏览器基线为 Chrome/Edge 最新稳定版
- 摄像头协议 V1 以 RTSP 为主
- Python 版本与部署矩阵需统一定义，不保留多套冲突口径

#### FR-G05 可维护性

- 统一任务接口和领域命名
- 策略与模板能力合并，避免双轨
- 保留对旧接口的迁移说明

## 9. 标准 JSON 输出要求

### 9.1 通用要求

- 正式任务的模型返回必须是可解析 JSON
- 每个策略都应绑定一份 JSON Schema
- 系统必须保留模型原始响应，用于排障和评估

### 9.2 推荐通用结构

```json
{
  "result": "normal",
  "summary": "现场人员佩戴安全帽规范，未发现明显异常。",
  "severity": "low",
  "has_violation": false,
  "issues": [],
  "evidence": [],
  "suggestions": []
}
```

### 9.3 推荐 Schema 示例

```json
{
  "type": "object",
  "required": ["result", "summary", "severity", "has_violation", "issues", "evidence", "suggestions"],
  "properties": {
    "result": {
      "type": "string"
    },
    "summary": {
      "type": "string"
    },
    "severity": {
      "type": "string",
      "enum": ["low", "medium", "high"]
    },
    "has_violation": {
      "type": "boolean"
    },
    "issues": {
      "type": "array",
      "items": {
        "type": "string"
      }
    },
    "evidence": {
      "type": "array",
      "items": {
        "type": "string"
      }
    },
    "suggestions": {
      "type": "array",
      "items": {
        "type": "string"
      }
    }
  }
}
```

## 10. 模型评估与选型

### 10.1 目标

针对视觉识别任务，系统需要具备对比不同模型提供方能力的评估机制，用于模型选型和迁移决策。

### 10.2 评估对象

- 智谱：当前基线提供方
- OpenAI：重点迁移评估对象

### 10.3 样本集要求

样本集至少覆盖以下场景：

- 安全帽佩戴
- 火情识别
- 日期/OCR 识别
- 小目标识别
- 低光环境
- 模糊场景
- 遮挡场景

### 10.4 评估指标

| 指标 | 说明 |
|---|---|
| 准确率 | 与人工标注比对的准确程度 |
| 稳定性 | 多次请求结果一致性 |
| 结构化成功率 | 符合 JSON Schema 的比例 |
| 平均响应时延 | 完成一次任务的平均耗时 |
| 单次成本 | 模型调用成本 |

### 10.5 迁移原则

迁移采用渐进式策略：

1. 抽象统一的模型提供方适配层
2. 保留智谱作为当前基线
3. 引入 OpenAI 作为对照提供方
4. 通过真实样本评估得出选型结论
5. 在灰度环境中逐步切换主模型

### 10.6 默认建议

以下建议是实现侧默认建议，不应在产品文档中写成唯一承诺：

- 精度优先候选：`gpt-5.4`
- 成本/时延对照候选：`gpt-5-mini`
- 输出方式优先采用 `json_schema`
- 新接入应优先考虑 OpenAI 当前官方的 Models 页面和 Responses API 指南

### 10.7 历史名词说明

“GPT-4V”仅作为历史参考称呼，不作为新接入目标名。新方案以当前官方可用视觉模型与官方文档为准。

## 11. 数据与报表指标定义

### 11.1 基础指标

| 指标 | 定义 |
|---|---|
| 任务总数 | 指定时间范围内创建的任务总数 |
| 完成率 | `completed / 全部任务` |
| 失败率 | `failed / 全部任务` |
| 结构化成功率 | Schema 校验通过的记录占比 |
| 异常比例 | `has_violation = true` 的记录占比 |
| 策略使用频率 | 各策略对应任务数占比 |

### 11.2 反馈类指标

| 指标 | 定义 |
|---|---|
| 已复核率 | 已完成人工反馈的记录占比 |
| 人工确认准确率 | 被标记为 `correct` 的记录占已复核记录的比例 |
| 异常误报率 | 被模型判异常但人工判定错误的比例 |

### 11.3 指标口径约束

- 任何“准确率”指标必须基于人工反馈
- 未经人工复核的记录不得用于准确率口径
- 聚合统计口径必须与任务记录导出字段保持一致

## 15. 验收标准

### 15.1 功能验收

- 上传单张和批量图片后，系统能够成功创建任务并生成任务记录
- 摄像头单次抽帧和定时抽帧能够成功触发任务
- 分析策略启用、禁用、编辑后能够影响新任务，不影响历史记录
- 模型返回非法 JSON 时，系统能标记结构化失败并保存原始响应
- 任务支持取消、失败展示和状态追踪
- 历史记录支持查询、详情查看和 CSV 导出
- 人工反馈标记后，统计口径能够反映复核结果

### 15.2 数据验收

- 每条记录都保留输入图片、策略快照、模型原始响应、标准 JSON
- 记录详情字段与导出字段保持一致
- 基础看板口径与记录聚合口径一致

### 15.3 非功能验收

- 摄像头连接失败时，页面能看到明确错误
- 模型密钥在接口和 UI 中保持脱敏
- 文件访问接口不能越权访问任意路径
- 高频提交任务时，队列状态和失败信息仍可追踪

### 15.4 模型评估验收

- 同一标准样本集能够生成 OpenAI 与智谱的对照结果
- 至少输出准确率、结构化成功率、平均耗时、单次成本
- 输出迁移建议结论：继续双栈、切主 OpenAI、保留智谱主栈三选一
