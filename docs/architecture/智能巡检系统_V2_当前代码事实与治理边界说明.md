# 智能巡检系统 V2 当前代码事实与治理边界说明

## 1. 文档目的

本文档用于澄清 V2 主线在当前仓库中的真实代码边界、事实源和治理重点，避免继续直接引用已经部分过时的历史综合分析结论。

本文档优先级低于需求基线和实施基线，但高于历史分析结论。涉及当前代码实现、验证脚本和交付入口时，以本文档列出的事实源为准。

## 2. 当前主事实源

当前仓库的主事实源固定为以下四类：

1. `docs/product/智能巡检系统_需求规格与功能更新方案_v2.md`
2. `docs/plan/智能巡检系统_V2_实施计划与功能设计方案.md`
3. `backend-v2` / `frontend-v2` / `local-detector` / `scripts/v2` 中的当前代码结构
4. `Makefile`、`README_V2.md` 和 `scripts/v2/*.sh` 中的当前验证与交付入口

历史综合分析报告仍保留，但仅作为阶段性治理参考，不再单独作为当前架构优先级的唯一依据。

- `mes-workflow-designer`
- 根目录旧入口 `main.py`
- `workflow_prototype`
- 旧飞书/Aily 相关残留脚本与资料

上述内容可保留作为历史参考，但不进入 V2 主交付范围，不作为本轮治理拆分目标。

## 4. 当前代码事实

### 4.1 前端

- 权限规则已经收口到 `frontend-v2/src/shared/auth/permissions.ts`
- 路由守卫和菜单展示均消费同一份导航/角色定义
- `shared/api/configCenter.ts` 已拆为按域导出，不再是单文件巨石

这意味着历史“前端权限分散”和“configCenter 巨石”结论已经部分失效。后续治理应以“继续防止跨域聚合回潮”为主，而不是重复执行已完成的拆分工作。

### 4.2 后端

- `task_dispatcher` 是异步 worker 调度唯一入口
- `scheduler` 运行入口保留在 `app/schedulers/runner.py`
- 调度逻辑已按职责拆为：
  - `schedule_dispatch_service`
  - `camera_status_sweep_service`
  - `signal_monitor_orchestrator`
  - `alert_delivery_service`
- 任务逻辑已按职责拆为：
  - `job_queries`
  - `job_creation_service`
  - `job_command_service`
- `job_service` / `scheduler_service` 当前仅承担兼容 facade 角色
- `backend-v2/tests/test_service_module_boundaries.py` 已锁定以下边界：
  - API routes 不得回退为直接依赖 `job_service` / `scheduler_service`
  - `app/schedulers/runner.py` 只能直接组合拆分后的调度服务
  - 两个 facade 必须保持为纯 re-export shim，不能重新长出业务逻辑

### 4.3 验证与交付

- `preflight.sh` 是当前真实异步栈验证基线，会启动 API / worker / scheduler / 依赖并输出 readiness 证据
- `real-async-test.sh` 是明确的“真实异步集成验证”入口，语义上区别于 SQLite + `CELERY_ENABLED=false` 的轻量测试基座
- 默认生产发布口径为 `make v2-release-gate-final`
- `release-gate.sh` 中使用 override/bypass 标志的运行会在摘要产物中显式标记为 bypass run
- `2026-04-12` 已完成一次无 override 的 `make v2-release-gate-final`，最新通过产物位于 `data/release-gates/release-gate-20260412-213844-49f760fc/`

### 4.4 旧数据回灌基线

- `data/surveillance.db` 不是独立可用的唯一旧数据源，它与 `data/detections/` 下的历史样本文件共同构成 `reconcile` / `release-drill` 的事实基线
- 当前至少有 13 个样本文件是发布门禁必需 fixture，删除这些文件会直接导致 `make v2-reconcile` 和 `make v2-release-gate-final` 失败
- 这些 fixture 已在 `.gitignore` 中显式放行，并由 `backend-v2/tests/test_v2_script_contracts.py` 的完整性测试保护
- 后续若要替换或移除这批文件，必须同步更新 `surveillance.db`、回灌逻辑和门禁基线，不能只做文件清理

## 5. 当前治理重点

本轮治理只聚焦以下事项：

1. 后端编排边界清晰化
2. 真实异步链路验证与发布门禁收紧
3. 文档与代码事实对齐

本轮不做以下事项：

1. 不扩新业务功能
2. 不修改现有 REST 路由契约
3. 不做前端信息架构和视觉重构
4. 不做数据库大改或微服务拆分

## 6. 使用建议

- 分析当前架构时，先读本文档，再读历史综合分析报告
- 做 V2 发布或回归时，优先使用：
  - `make v2-real-async-test`
  - `make v2-preflight`
  - `make v2-release-gate-final`
- 只有在非正式演练、补证据或人工兜底场景下，才允许使用 `release-gate.sh` 的 override/bypass 参数，并必须写明原因
