# 智能巡检系统 V2 后续优化清单（2026-04-12）

## 范围说明
- 本清单只记录本轮提交后仍建议继续优化的事项。
- 优先级以 `P0 > P1 > P2` 标识，默认以不扩大外部联调面为原则。

## 优化项

### P0：发布与运行稳定性
- 统一脚本产物目录治理：持续补齐 `data/*` 运行产物的忽略规则，避免验证日志反复污染工作区。
- 明确“可发布入口”与“演练入口”差异：继续强化 `release-gate-final` 作为唯一正式口径，旁路运行必须有 reason 和产物标记。
- 为关键脚本补失败注入验证：至少覆盖依赖未就绪、队列不可用、worker 未注册等高频失败场景。

### P1：后端编排与模型调用
- 将 `version_recognition` 从“路由 + 执行细节”再下沉为清晰服务边界，降低 `job_execution_service` 体积和职责混杂。
- 补充 OCR/版本提取链路的契约测试：覆盖 ROI 回退、旋转回退、放大回退和空结果降级路径。
- 抽象统一的 Provider 重试策略：`OpenAIAdapter` 现有重试逻辑可沉淀为通用策略模块，避免各适配器重复实现。

### P1：前端状态与 E2E 稳定性
- 对 camera 页面 state hooks 做依赖方向收敛，减少 URL、表单、监控规则之间的双向耦合。
- E2E 用例按“业务主链路/观测类/回归类”分组，降低单次回归的偶发失败定位成本。
- 将 URL 同步与查询态测试继续细化到 hook 级别，减少只靠端到端才能发现的问题。

### P2：工程治理与文档一致性
- 对新增部署脚本与说明文档做“V2主线/历史兼容”标注，避免与旧入口混用。
- 建立“架构事实文档”定期校对节奏，确保文档结论与代码现状不漂移。
- 增加提交前检查清单：改动分类、联调影响、验证证据路径、是否涉及运行产物。

## 建议执行顺序
1. 先做发布与脚本稳定性（P0）。
2. 再做后端编排边界和模型调用治理（P1）。
3. 最后收敛前端 state 与 E2E 稳定性（P1），并补充文档和流程治理（P2）。
