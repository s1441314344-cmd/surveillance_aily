## 1. 文档目标

本文档将 V2 需求和实施方案转换为可执行 backlog，并明确：

- Epic / Story / Task 的拆分方式
- 每项工作的优先级、依赖和责任角色
- Phase 1 可直接启动的工程骨架范围
- 每项任务的完成定义与验收标准

---

## 2. Backlog 组织规则

### 2.1 优先级定义

- `P0`：阻断 V1 核心闭环
- `P1`：影响上线质量或核心效率
- `P2`：增强项或后续演进项

### 2.2 状态定义

- `Todo`
- `In Progress`
- `Blocked`
- `Review`
- `Done`

### 2.3 责任角色简称

- `PM`：产品经理
- `ARCH`：技术负责人/架构师
- `FE`：前端开发
- `BE`：后端开发
- `AI`：AI/算法开发
- `QA`：测试工程师
- `OPS`：运维/DevOps

### 2.4 完成定义（关闭标准）

每项 Backlog 关闭前必须同时满足：

- 变更已合入仓库（代码或文档），且变更范围与 Backlog 描述一致。
- 对应接口/数据结构/页面行为已在文档中更新，并可被追溯。
- 验收条目已关联到测试入口（手工步骤或自动化脚本）。
- 自动化用例或验证脚本至少通过一次，并保留执行记录。
- 证据产物已归档到仓库约定路径，推荐模式：
  - `artifacts/acceptance/<yyyy-mm-dd>/<backlog-id>/screenshots/`
  - `artifacts/acceptance/<yyyy-mm-dd>/<backlog-id>/logs/`
  - `artifacts/acceptance/<yyyy-mm-dd>/<backlog-id>/reports/`
- 追踪矩阵中对应行（FR -> Backlog -> 验收条目 -> 自动化 -> 证据）已补齐，无空值。
- 不存在阻断下游推进的 P0/P1 缺陷。

### 2.5 冻结条件

Backlog 基线冻结并停止新增 P0 条目，需满足：

- 所有 P0 条目状态为 `Done` 或经 PM+ARCH 书面降级。
- `docs/testing/智能巡检系统_V2_全量验收清单.md` 的 P0 核心主线全部通过。
- `docs/plan/智能巡检系统_V2_需求到验证追踪矩阵.md` 完成评审并与当前仓库一致。

## 8. 最终完整测试方案

系统测试阶段必须至少覆盖以下主流程：

### 主流程

1. 登录 -> 创建策略 -> 上传单张图片 -> 查看记录 -> 提交反馈 -> 看板刷新
2. 登录 -> 创建策略 -> 上传批量图片 -> 查看任务状态汇总 -> 查看多条记录
3. 登录 -> 配置摄像头 -> 创建单次摄像头任务 -> 查看记录
4. 登录 -> 配置摄像头 -> 创建定时任务 -> 调度触发 -> 生成记录 -> 查看看板变化

### 异常流程

- 非法图片
- 模型返回非 JSON
- JSON 不符合 Schema
- 摄像头断连
- 任务取消
- 权限不足

### 测试层次

- 单元测试
- 集成测试
- 系统测试
- E2E 测试
- 长时稳定性测试
