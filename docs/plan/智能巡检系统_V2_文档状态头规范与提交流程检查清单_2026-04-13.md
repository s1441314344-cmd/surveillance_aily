# 智能巡检系统 V2 文档状态头规范与提交流程检查清单

文档定位: V2主线  
事实基线日期: 2026-04-13  
默认发布入口: `make v2-release-gate-final`  
是否允许旁路: 否（如需旁路，必须提供 reason + 产物标记）  
替代/被替代关系: 无  
Owner: Thread D（P2治理）+ Supervisor

---

## 1. 适用范围

本规范用于 V2 文档治理最小基线，覆盖：

1. 文档状态头统一写法
2. `V2主线 / 历史兼容 / 历史记录` 标注规则
3. 提交前检查清单（Pre-merge Checklist）
4. 监督者可核查的触发条件与责任分工

## 2. 文档状态头规范（必填）

所有新增或更新的治理/发布相关文档，文首必须包含以下状态头：

```md
文档定位: [V2主线 | 历史兼容 | 历史记录]
事实基线日期: YYYY-MM-DD
默认发布入口: make v2-release-gate-final
是否允许旁路: 否（若是，必须写明 reason + 产物标记）
替代/被替代关系: <doc-path 或 无>
Owner: <角色或责任人>
```

## 3. 三类标注规则

### 3.1 V2主线

1. 可直接指导当前开发、验证与发布。
2. 必须给出可复查证据路径（如 `data/preflight-logs/.../summary.json`、`data/release-gates/.../gate-summary.json`）。
3. 与其他文档冲突时，优先级最高。

### 3.2 历史兼容

1. 可用于理解演进背景或兼容约束。
2. 不得覆盖 `V2主线` 的当前发布口径与事实结论。
3. 引用时必须明确“仅供兼容参考”。

### 3.3 历史记录

1. 仅用于过程留档（提审单、旧批次复验单、旧发布说明等）。
2. 不得作为当前状态判定依据。
3. 若被引用，必须配套给出最新 `V2主线` 文档链接。

## 4. 提交流程检查清单（Pre-merge Checklist）

```md
# Pre-merge Checklist (V2)

- [ ] 改动分类已声明：P0 / P1 / P2（可多选）
- [ ] 联调影响已声明：无 / API / Worker / Scheduler / Frontend
- [ ] 验证命令与结果已给出（至少一条）
- [ ] 证据路径已给出（preflight/release-gate 摘要或测试产物）
- [ ] 若使用 bypass/override：已写明 reason，且产物有标记
- [ ] 若涉及 data 基线：已说明是否影响 release baseline fixture
- [ ] 文档标注已更新：V2主线 / 历史兼容 / 历史记录
- [ ] 未将 2026-04-04 文档当作当前事实引用
```

## 5. 监督可核查简表（规则、触发条件、责任人）

| 编号 | 规则 | 触发条件 | 责任人 |
| --- | --- | --- | --- |
| R1 | 文首必须有状态头 | 新增或更新治理/发布相关文档 | 文档提交人 |
| R2 | 必须标注文档定位（三选一） | 文档进入评审或提审 | 文档提交人 |
| R3 | `V2主线` 必须附证据路径 | 声明“当前事实”或“当前可发布” | 对应线程 Owner（P0/P1/P2） |
| R4 | 历史文档不可替代当前结论 | 引用 `2026-04-04` 批次文档 | 引用人 + Reviewer |
| R5 | 发布口径统一为 `v2-release-gate-final` | 涉及发布门禁、旁路、release 结论 | P0 Owner + Supervisor |
| R6 | 旁路运行必须有 reason 与标记 | 出现 `skip` / `override` / bypass | 发起人 + Supervisor |
| R7 | Pre-merge Checklist 必须完整 | 创建 PR 或提审单 | PR 作者 |
| R8 | 跨线程越界改动直接阻断 | 发现变更超出线程白名单 | Supervisor |

## 6. 执行节奏（最小）

1. 每周一 10:00：文档一致性校对（主线文档、README_V2、发布口径）。
2. PR 创建前：提交人自检 `Pre-merge Checklist`。
3. PR 合并前：Supervisor 复核 `R1-R8`，不通过则退回。
