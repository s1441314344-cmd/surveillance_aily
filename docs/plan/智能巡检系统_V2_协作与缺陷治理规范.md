# 智能巡检系统 V2 协作与缺陷治理规范

## 1. 目标

本规范用于约束 V2 在并行开发、验证、提审、缺陷关闭阶段的最小治理要求，避免：

- 多人共享 `data/` 产物导致结论污染
- 脚本通过“最新文件”回溯发布依据
- 缺陷修复后没有复验闭环
- 提审缺少影响面、证据、风险与回滚说明

## 2. 分支与工作区

### 2.1 分支策略

- 主线分支仅接受已通过 release gate 的变更。
- 功能、治理、修复分支统一使用 `codex/<topic>` 或团队约定前缀。
- P0/P1 阶段默认冻结新功能，只允许提交与基线、验证、门禁、治理直接相关的改动。

### 2.2 worktree 使用要求

- 并行开发必须使用独立 worktree 或独立本地目录，不允许多人共用同一工作目录验证。
- 每个 worktree 必须拥有独立 `.env`、前端端口、输出目录。
- 如需并行起本地服务，必须显式设置：
  - `V2_ENVIRONMENT`
  - `V2_RUN_ID`
  - 独立 `--output-dir`

## 3. 验证实例隔离

- 验收、预演、回归产物必须按 `run_id + git_sha` 唯一定位。
- 发布与验收脚本禁止以“最新文件”作为默认输入；必须显式传入摘要或报告路径。
- 允许的证据目录模式：
  - `data/<stage>-logs/<run_id>/`
  - `artifacts/acceptance/<yyyy-mm-dd>/<run_id>/`

## 4. `data/` 产物治理

- `data/` 下所有结论性产物必须记录来源分支、`git_sha`、执行时间和执行人。
- 临时调试文件放入 `data/tmp/<run_id>/`，不得与验收产物混放。
- 需要长期保留的报告必须迁移到 `artifacts/acceptance/...` 或评审材料目录。

## 5. 缺陷生命周期

统一状态流转为：

`Open -> In Progress -> Fixed -> Retest -> Closed / Rejected`

### 5.1 必填字段

- 缺陷 ID
- 严重级别：`P0 / P1 / P2 / P3`
- 责任人
- 复验人
- 证据路径
- 影响面
- 风险与回滚说明

### 5.2 关闭标准

- `Fixed` 仅表示代码或配置已提交，不代表问题关闭。
- 只有复验通过且证据齐全，状态才可从 `Retest` 进入 `Closed`。
- 无法复现、非缺陷或需求变更项，才允许进入 `Rejected`，且必须记录原因。

## 6. 提审要求

每次提审必须附一页式材料，至少包含：

- 变更摘要
- 影响面
- 测试范围与结果
- 证据路径
- 已知风险
- 回滚方式

若以上任一项缺失，评审可直接退回，不进入发布门禁。
