## 1. 文档目标

本文档用于将 V2 实施文档进一步转换为项目执行方案，重点解决四类问题：

- 技术架构如何从现有系统演进为 V2 正式架构

## 2. 架构搭建总方案

## 2.1 目标架构

V2 正式架构采用：

- 前端：React + TypeScript + Vite + Ant Design
- 后端：FastAPI + PostgreSQL + Redis + Celery
- 调度：APScheduler
- 文件存储：本地文件系统抽象，预留 MinIO/S3 扩展
- 模型执行：Python provider adapter，支持 OpenAI 与智谱

架构目标是将当前“Flask + 静态页 + SQLite + 同步检测”演进为“模块化前后端 + 异步任务队列 + 记录中心 + 反馈中心 + 报表中心”的正式平台。

## 2.2 当前系统主要问题

| 类别 | 当前问题 | V2 整改方向 |
|---|---|---|
| 前端 | 单页 HTML 控制台过大，页面和状态耦合 | 重构为模块化 React SPA |
| 后端 | Flask 路由集中、领域边界混杂 | 重构为 FastAPI 分域模块 |
| 数据 | SQLite 适合轻量单机，不利于任务化与多角色协作 | 切换 PostgreSQL |
| 任务 | 同步接口、重复任务接口、定时任务模型不统一 | 统一为 `jobs + job_schedules + task_records` |
| 规则 | `rules + prompt_templates` 双轨并存 | 合并为 `strategies` |
| 安全 | 缺少正式鉴权、密钥暴露风险 | 增加 JWT、RBAC、脱敏和鉴权文件访问 |
| 测试 | 测试脚本零散，缺少系统化测试链路 | 建立单元、集成、系统、E2E 全链路 |

## 2.3 架构搭建工作流

V2 架构搭建按四条并行主线推进：

1. 前端平台主线
2. 后端平台主线
3. 数据与基础设施主线
4. 模型与任务执行主线

四条主线的交汇点是统一的 API 契约、领域模型和任务状态机。

## 3. 功能清单整改方案

## 3.1 整改原则

- 以 V2 需求文档为唯一基线
- 不再继续扩展旧版 Feishu/Aily 主流程
- 所有正式功能都要有明确实体、接口和页面入口
- 所有任务型能力都必须进入任务队列与任务记录体系

## 5. 联调整合方案

任务完成后的联调必须按以下顺序执行，避免多模块同时联调导致定位困难。

## 5.1 联调顺序

### Step 1 认证与基础设施联调

- 验证登录、Token 刷新、权限拦截
- 验证前端 API client、全局错误处理、后端鉴权中间件

### Step 2 配置中心联调

- 联调模型提供方管理
- 联调策略中心
- 联调摄像头 CRUD 与状态检查

### Step 3 任务中心联调

- 联调上传任务
- 联调摄像头单次任务
- 联调定时任务与计划启停
- 联调任务取消与状态刷新

### Step 4 记录与反馈联调

- 联调任务记录列表
- 联调记录详情
- 联调 CSV 导出
- 联调人工反馈

### Step 5 看板联调

- 联调汇总 KPI
- 联调趋势图
- 联调异常案例列表与跳转

## 5.2 联调输入输出清单

每次联调都必须明确：

- 输入：前端请求参数、测试账号、样本图片、摄像头/视频源
- 输出：数据库记录、任务状态、返回 JSON、页面展示结果
- 判定标准：是否满足接口契约与 UI 预期

## 5.3 联调责任

- 前后端接口问题：前端开发 + 后端开发共同处理
- 推理与结构化输出问题：后端开发 + AI/算法开发共同处理
- 用例覆盖与回归确认：测试工程师负责
- 优先级裁定和是否阻塞版本：产品经理 + 技术负责人共同决策

## 6. 功能单元验证方案

功能单元验证用于在系统联调前后，对每个模块独立做确认，避免“看起来能跑但局部失效”。

## 6.1 模块级验证清单

| 模块 | 验证重点 | 验证方式 | 责任角色 |
|---|---|---|---|
| 认证中心 | 登录、刷新、权限控制 | API 测试 + 页面验证 | 后端 + 前端 + 测试 |
| 模型提供方 | 配置保存、脱敏显示 | API 测试 + 表单验证 | 后端 + 测试 |
| 策略中心 | 策略 CRUD、Schema 校验 | 单元测试 + 页面操作 | 前端 + 后端 + AI |
| 摄像头中心 | CRUD、状态检查、错误回显 | API 测试 + mock 流验证 | 后端 + AI + 测试 |
| 任务中心 | 创建、取消、状态流转 | 集成测试 + UI 验证 | 前端 + 后端 + AI |
| 记录中心 | 查询、详情、导出 | API 测试 + 页面验证 | 前端 + 后端 + 测试 |
| 反馈中心 | 正确/错误标记、修正标签 | API 测试 + 页面验证 | 前端 + 后端 + 测试 |
| 报表中心 | 指标聚合、图表展示 | 聚合校验 + 页面验证 | 后端 + 前端 + 测试 |

## 6.2 单元验证出口标准

- 接口返回结构与契约一致
- 关键字段有边界值校验
- 失败场景有明确错误
- 页面交互没有阻断性问题
- 记录和状态变化能正确落库

## 7. 最终完整测试方案

最终完整测试用于确认系统在真实工作流下满足 V2 需求规格。

## 7.1 测试范围

### 全流程场景

1. 登录 -> 创建策略 -> 配置摄像头 -> 创建上传任务 -> 查看记录 -> 人工反馈 -> 看板刷新
2. 登录 -> 创建策略 -> 配置摄像头 -> 创建单次摄像头任务 -> 查看记录
3. 登录 -> 创建定时任务 -> 调度触发 -> 生成记录 -> 反馈 -> 看板统计
4. 上传批量图片 -> 多条记录生成 -> 部分失败 -> 任务状态正确汇总

### 异常场景

- 非法图片格式
- 模型返回非 JSON
- JSON 不符合 Schema
- 摄像头无法连接
- 摄像头抓帧失败
- 队列任务取消
- 反馈重复提交
- 导出大批量记录

### 权限场景

- 非管理员无法修改模型密钥
- 非策略配置员无法修改策略
- 分析查看者仅可读

## 7.2 测试层次

### 单元测试

- provider adapter
- schema validator
- 状态机
- 聚合逻辑
- 前端关键组件

### 集成测试

- FastAPI + PostgreSQL + Redis + Celery
- 文件存储
- RTSP mock 流
- provider contract test

### 系统测试

- 浏览器端到端测试
- 多角色权限测试
- 全流程业务测试
- 长时稳定性测试

## 7.3 测试出口标准

- 四条主流程全部通过
- P0/P1 缺陷全部关闭
- 没有阻断上线的安全问题
- 长时运行测试通过
- 看板指标与记录数据口径一致

## 8. 协作机制与交付规则

## 8.1 工作节奏

- 每周一：计划更新与风险同步
- 每周三：接口联调例会
- 每周五：阶段 demo 与缺陷评审

## 8.2 缺陷优先级

- `P0`：阻断核心流程或上线
- `P1`：核心功能明显异常但有替代路径
- `P2`：非核心问题或体验问题

## 8.3 完成定义

每项任务关闭前必须满足：

- 代码已合并到对应分支
- 单元测试通过
- 文档与接口契约更新
- 测试用例已补充
- 联调记录已确认

## 9. 默认假设

- V1 只覆盖核心闭环，不纳入 ONVIF、拖拽式看板、飞书/Aily 主流程恢复
- 团队按“1 名技术负责人 + 1 名前端 + 2 名后端/AI + 0.5 名 QA + 0.5 名 DevOps”配置
- 旧 Flask 系统在 V2 切换前保持只修不扩
- PostgreSQL 为 V2 正式数据库，SQLite 仅作为历史数据源
- OpenAI 候选模型名称在 Phase 0 启动周以官方文档再次确认
