# 智能巡检系统 V2 P0-P3 整体优化 PR 评审说明

## 标题

`V2 治理收敛与架构边界优化（P0-P3）`

## 背景

当前 V2 主线已经具备产品骨架，但仍存在以下阻断问题：

- 需求、实施、README 技术口径不一致
- 验收项无法追踪到自动化证据
- 发布门禁对 summary/report 产物依赖不严
- 后端 service 直接依赖 worker，边界易继续退化
- 前端 Router / Layout 权限规则重复
- `configCenter.ts` 过大且跨域耦合
- 提审与复验材料缺失，治理闭环不完整

本 PR 的目标不是扩功能，而是完成一次主线治理收敛。

## 本次改动

### 1. 基线与治理

- 统一 V2 基线文档与状态字段
- 新增需求到验证追踪矩阵
- 收紧 Backlog 关闭标准与缺陷治理规则
- 补齐提审模板与验收清单映射

### 2. 发布与门禁

- 为 `preflight` / `uat` / `verify` / `release-gate` 增加完整 run metadata
- 强制显式 summary/report 输入，移除“最新文件”依赖
- 加入 `security` 与 `reconcile` 最小发布验证入口
- 收紧 override 使用规则并记录原因

### 3. 后端边界治理

- 引入 `task_dispatcher`
- `job_service` / `scheduler_service` 不再直接 import worker task
- signal monitor sweep 独立为 `scheduler_signal_monitor_sweep_service`
- `alert_service` 拆为 facade + helper 结构

### 4. 前端边界治理

- 用 `permissions.ts` 作为单一权限与导航定义源
- Router 与 Layout 共用一套角色规则
- `configCenter.ts` 拆成 domain modules，并保留兼容 barrel
- 增强 barrel runtime/type compatibility 覆盖

### 5. 评审与复验材料

- 新增提审单
- 新增发布说明
- 新增复验记录单

## 不包含内容

- 不修改外部 HTTP API
- 不新增业务功能
- 不做数据库 schema/migration 变更
- 不处理与本轮无关的仓库文档重组内容

## 风险

- `alert_service` 未来扩展时必须继续走 facade 注入，否则 helper 与 facade 行为可能漂移
- `configCenter` 对仓库内调用兼容，但仓库外若存在消费者仍建议做一次集成联调
- 当前 Node 环境存在 `--localstorage-file` warning，但已通过测试 mock 消除功能性影响

风险等级：`P2`

## 建议评审顺序

1. 文档治理与追踪矩阵
2. 脚本与门禁
3. 后端派发与告警边界
4. 前端权限与 API barrel
5. 提审 / 发布 / 复验材料

## 评审关注点

- 是否仍保持外部 API 兼容
- release gate 是否已完全去掉“最新文件”依赖
- 后端 service 是否真正摆脱 worker 反向依赖
- 前端菜单与路由权限是否完全一致
- 发布/提审/复验材料是否足以进入门禁
