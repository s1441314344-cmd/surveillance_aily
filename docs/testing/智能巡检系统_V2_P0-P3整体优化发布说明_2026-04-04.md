# 智能巡检系统 V2 P0-P3 整体优化发布说明

## 1. 发布目标

本次发布不新增业务功能，目标是将 V2 从“可运行骨架”收敛为“可验证、可放行、可持续演进”的主线版本，重点解决：

- 基线文档与实施口径不一致
- 需求到验证证据链不闭合
- 发布门禁可被弱绕过
- 后端服务边界退化、直接依赖 worker 实现
- 前端权限与 API 聚合层耦合过高
- 提审、复验、缺陷闭环材料不足

## 2. 发布范围

### P0 基线与追踪矩阵

- 统一 `README_V2.md`、需求、计划、Backlog、验收清单中的 V2 技术基线与状态字段
- 新增正式追踪矩阵：`FR -> Backlog -> 验收 -> 自动化测试 -> 证据`
- 收紧 Backlog 关闭标准，禁止“测试待补”直接关闭 P0/P1 项

### P1 验证链路与门禁

- 为 `preflight`、`uat`、`verify`、`release-gate`、`security`、`reconcile` 增加统一元数据输出
- `verify` 强制检查前置摘要存在
- `release-gate` 收紧绕过条件，显式记录 override 理由
- `release-checklist` 与 `release-gate` 禁止依赖“最新文件”回溯发布产物

### P2 架构边界治理

- 后端：
  - 引入 `task_dispatcher`
  - `job_service`、`scheduler_service` 不再直接依赖 worker task
  - `scheduler_signal_monitor_sweep_service` 抽离 signal monitor sweep 责任
  - `alert_service` 拆为 facade + event/webhook/notification helper 模块
- 前端：
  - `permissions.ts` 统一 Router / Layout 权限和导航来源
  - `configCenter.ts` 拆为 domain modules，并保留兼容 barrel

### P3 协作与闭环治理

- 新增协作与缺陷治理规范
- 新增提审模板与本次已填写提审单
- 补齐测试环境 `localStorage` mock，保证当前环境前端全量测试可执行

### 重点回归

- 派发端口解耦回归

- alert facade passthrough / fan-out 回归
- 前端权限统一回归
- `configCenter` barrel compatibility 回归

## 5. 已知注意事项

- 当前 Node 环境会打印 `--localstorage-file was provided without a valid path` warning，但不影响测试结论。
- `configCenter` 已保持仓库内兼容；若存在仓库外消费者，建议在集成环境补一次真实联调。
- `alert_service` 未来若继续扩展 monkeypatch 点位，必须沿 facade 注入方式补齐测试，否则易出现 helper/facade 漂移。

## 6. 发布建议

- 建议作为一次“治理收敛版”发布，不与业务功能发布混合。
- 建议发布前由 QA Lead 与技术负责人联合完成一次复验签字。
- 建议发布后第一时间观察：
  - scheduler 调度任务是否正常入队
  - alert webhook/通知搜索是否正常
  - 菜单权限与页面访问是否一致
  - release gate 输出摘要字段是否完整

## 7. 回滚建议

- 如出现任务派发、signal monitor、告警通知或前端权限异常，按提审单中的“变更组”整体回滚，不建议只回滚局部 helper 文件。
- 回滚后必须重新执行：
  - `PYTEST_DISABLE_PLUGIN_AUTOLOAD=1 pytest backend-v2/tests -q`
  - `npm test`
  - `npm run build`
